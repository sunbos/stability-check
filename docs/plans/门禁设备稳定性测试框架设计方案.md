# 门禁设备稳定性测试框架设计方案
## 基于 pytest + stability-harness-loop-multiagent · 支持动态配置

---

## 1. 文档概述
### 1.1 背景
针对海康门禁设备的HTTP接口场景，需实现无人值守的自动化稳定性测试，核心验证设备在反复操作、重启循环下的功能可用性、事件完整性与运行可靠性。测试需支持自定义循环规则、动态参数调整，适配前端通过环境变量传参的调用方式，并具备批量扩展多类稳定性场景的能力。

### 1.2 核心目标
1. 落地「开门+重启循环」MVP稳定性测试，支持1000轮及以上无人值守自动执行
2. 采用「YAML基准配置 + 环境变量动态覆写」的配置体系，兼容前端 `os.env` 传参模式
3. 基于通用稳定性测试模板构建可复用架构，核心能力与业务逻辑解耦
4. 依托pytest生态实现用例管控、断言体系与报告输出，保障测试标准化
5. 预留充足扩展能力，支持后续快速新增多场景、多设备稳定性测试用例

### 1.3 适用范围
- 单台/多台海康门禁设备HTTP接口稳定性测试
- 循环操作类、长稳运行类、故障恢复类稳定性测试场景
- 前端动态传参、CI流水线调度、本地调试等多种执行方式
- 可扩展至其他支持网络协议调用的安防设备、后端服务稳定性测试

---

## 2. 整体架构设计
整体采用**三层解耦架构**，遵循「通用能力下沉、业务逻辑插拔、用例轻量编排」的设计原则，核心复用 `stability-harness-loop-multiagent` 模板的多Agent范式，以pytest作为统一执行入口。

### 2.1 三层架构分层
| 架构层级 | 对应目录 | 核心职责 | 业务耦合度 |
|----------|----------|----------|------------|
| 核心框架层 | `stability_harness_loop_multiagent/` | 通用循环引擎、多Agent基类、动态配置中心、异常熔断、统计上报，与具体业务无关 | 0%（完全通用） |
| 业务适配层 | `adapters/` | 被测设备接口封装、各Agent业务实现、专属校验规则 | 100%（业务专属） |
| 用例执行层 | `tests/` | 基于pytest的测试用例编排、场景组合、结果断言、报告输出 | 低（仅做流程编排） |

### 2.2 多Agent协作范式
将稳定性测试的全链路职责拆分为4类独立Agent，各司其职，通过循环引擎统一调度：
- **Operator Agent（操作执行器）**：负责向被测设备下发业务指令，如开门、重启、参数配置
- **Probe Agent（探活检测器）**：负责轮询设备运行状态，判定在线/离线、服务就绪状态
- **Validator Agent（结果校验器）**：负责业务断言，如事件一致性校验、状态参数校验、数据完整性校验
- **Reporter Agent（统计上报器）**：负责全量日志记录、轮次统计、失败快照留存、最终报告生成

### 2.3 核心设计原则
1. **开闭原则**：新增测试场景/设备时，无需修改核心框架代码，仅需补充业务适配层
2. **单一职责**：各模块职责边界清晰，循环引擎不涉及业务逻辑，业务Agent不处理调度逻辑
3. **配置驱动**：所有阈值、次数、时间参数全部抽离配置，支持基准默认+动态覆写双层管理
4. **异常隔离**：单轮、单步骤异常不影响整体测试执行，具备熔断、兜底、断点续跑能力

## 3. 核心框架能力详解
### 3.1 循环引擎
循环引擎是框架的调度核心，负责全生命周期的测试流程管控，核心能力包括：
1. **双循环模式**
    - 按次数循环：配置总轮次，执行达标后自动终止
    - 按时长循环：配置总运行时长，超时后自动终止
2. **生命周期钩子**：预留「轮次前置、操作前置、探活前置、校验前置、轮次后置」钩子，支持插件式注入故障、监控等扩展能力
3. **执行控制**：支持轮次间隔配置、连续失败熔断、断点续跑、手动暂停/恢复

### 3.2 分层动态配置体系
采用多层级配置覆盖机制，兼顾YAML的结构化易维护性与环境变量的动态灵活性，完美适配前端 `os.env` 传参场景。

#### 3.2.1 配置优先级规则
优先级从高到低依次为：
> **pytest 命令行参数（可选扩展） > 系统环境变量（os.env） > YAML 基准配置 > 代码默认值**

- YAML基准配置：存放结构化固定参数与默认阈值，作为基础模板，无需频繁修改
- 环境变量：传入单次测试需动态调整的参数，仅覆盖指定字段，其余自动沿用YAML默认值
- 未设置任何动态参数时，完全等价于纯YAML配置，向后100%兼容

#### 3.2.2 环境变量命名映射规则
统一使用 `STABILITY_` 前缀避免全局冲突，嵌套字段以下划线连接，与YAML结构一一对应，示例如下：

| YAML 嵌套字段 | 对应环境变量 | 字段说明 |
|--------------|--------------|----------|
| `loop.total_rounds` | `STABILITY_LOOP_TOTAL_ROUNDS` | 测试总轮次 |
| `loop.mode` | `STABILITY_LOOP_MODE` | 循环模式（count/duration） |
| `loop.restart_max_timeout` | `STABILITY_LOOP_RESTART_MAX_TIMEOUT` | 重启最大超时时间（秒） |
| `loop.online_warmup_time` | `STABILITY_LOOP_ONLINE_WARMUP_TIME` | 设备上线后预热时长（秒） |
| `device.host` | `STABILITY_DEVICE_HOST` | 设备IP地址 |
| `event.time_offset` | `STABILITY_EVENT_TIME_OFFSET` | 事件查询时间偏移量（秒） |

#### 3.2.3 配置加载核心逻辑
封装通用配置加载器，先加载YAML基准配置，再递归读取匹配前缀的环境变量，自动完成类型转换后覆写对应字段，所有逻辑收敛在配置层，业务代码无感知。

```python
# stability_harness_loop_multiagent/config.py
import os
import yaml
from typing import Dict, Any

class ConfigLoader:
    def __init__(self, config_path: str, env_prefix: str = "STABILITY_"):
        self.config_path = config_path
        self.env_prefix = env_prefix

    def load(self) -> Dict[str, Any]:
        # 1. 加载YAML基准配置
        with open(self.config_path, "r", encoding="utf-8") as f:
            base_config = yaml.safe_load(f)

        # 2. 用环境变量递归覆盖对应字段
        self._override_by_env(base_config)
        return base_config

    def _override_by_env(self, config: Dict[str, Any], parent_key: str = ""):
        for key, value in config.items():
            current_key = f"{parent_key}{key.upper()}" if parent_key else key.upper()
            env_key = f"{self.env_prefix}{current_key}"

            if isinstance(value, dict):
                # 嵌套字典递归处理
                self._override_by_env(value, f"{current_key}_")
            elif env_key in os.environ:
                # 按原配置类型自动转换环境变量
                config[key] = self._type_convert(os.environ[env_key], type(value))

    @staticmethod
    def _type_convert(value: str, target_type: type) -> Any:
        if target_type == int:
            return int(value)
        elif target_type == float:
            return float(value)
        elif target_type == bool:
            return value.lower() in ("true", "1", "yes")
        return value
```

### 3.3 异常兜底与熔断机制
1. 全链路异常捕获：所有HTTP请求、业务逻辑均包裹异常处理，异常自动标记失败并留存现场
2. 单轮失败不中断：任意步骤失败均记录日志后自动进入下一轮，保障无人值守测试连续性
3. 连续失败熔断：连续失败达到配置阈值后自动终止测试，避免无意义空跑与设备异常放大
4. 超时统一管控：所有网络请求统一超时阈值，避免脚本卡死

### 4.4 单轮标准执行流程
每轮测试严格按顺序执行，所有步骤带毫秒级时间戳记录：
1. **前置基准校验**：调用状态接口确认设备在线，记录本轮起始时间戳
2. **下发开门指令**：调用远程开门接口，校验指令受理成功
3. **下发重启指令**：调用设备重启接口，校验指令受理成功
4. **重启静默期**：固定时长不发送任何请求，避开设备完全离线阶段
5. **轮询探活**：按间隔调用状态接口，连续2次返回在线判定重启成功，超时则判定失败
6. **上线预热**：设备在线后等待固定时长，保障内部服务与事件落库就绪
7. **状态复核**：校验设备核心运行参数与重启前一致
8. **事件校验**：按时间范围+事件类型筛选，校验开门、重启事件完整存在
9. **轮次收尾**：记录本轮结果，进入轮次间隔，准备下一轮

### 4.5 测试用例示例
```python
# tests/test_door_restart_stability.py
import pytest

class TestDoorRestartStability:
```

## 5. 动态配置实操指南
### 5.1 环境变量传参示例
前端通过 `os.env` 注入参数时，无需修改YAML文件，仅设置对应环境变量即可动态调整配置。
例如：动态设置总轮次为2000、重启最大超时为300秒：
```bash
# 前端注入环境变量
export STABILITY_LOOP_TOTAL_ROUNDS=2000
export STABILITY_LOOP_RESTART_MAX_TIMEOUT=300

# 执行pytest，自动合并动态参数
pytest tests/test_door_restart_stability.py -v
```

最终生效规则：
- `loop.total_rounds = 2000`（环境变量覆盖）
- `loop.restart_max_timeout = 300`（环境变量覆盖）
- 其余所有字段均沿用YAML文件中的默认值

### 5.2 命令行参数扩展（可选）
若需支持本地调试时手动传参，可叠加pytest命令行参数，优先级高于环境变量：
```python
# tests/conftest.py 追加
def pytest_addoption(parser):
    parser.addoption("--total-rounds", type=int, help="动态指定测试总轮次")

@pytest.fixture(scope="session")
def test_config(request):
    loader = ConfigLoader(config_path="configs/door_restart_stability.yaml")
    config = loader.load()
    
    # 命令行参数优先级最高
    cli_rounds = request.config.getoption("--total-rounds")
    if cli_rounds is not None:
        config["loop"]["total_rounds"] = cli_rounds
    return config
```

使用方式：
```bash
pytest tests/test_door_restart_stability.py --total-rounds=5000 -v
```

### 5.3 方案核心优势
1. **兼顾结构化与灵活性**：复杂固定配置由YAML统一管理，动态参数通过环境变量注入，适配前端调用、CI流水线、本地调试全场景
2. **向后完全兼容**：不设置任何环境变量时，与纯YAML配置行为完全一致
3. **粒度精准可控**：支持单字段精准覆盖，无需全量替换配置；新增字段自动支持环境变量覆写
4. **架构无侵入**：所有逻辑收敛在配置加载层，循环引擎、业务Agent、测试用例均无感知

```python
def validate_continuous_open_events(self, start_time, end_time, expected_count):
```

#### 步骤3：新增pytest测试用例

```python
import pytest
```

## 8. 开发与维护规范
### 8.1 分层边界规范
1. 业务逻辑仅写入`adapters`层，禁止修改核心框架层代码
2. 新增Agent必须继承对应基类，严格遵循基类方法签名规范
3. pytest用例层仅做场景编排与结果断言，不编写复杂业务逻辑

### 8.2 配置管理规范
1. 固定业务参数、默认阈值写入YAML基准配置，禁止在代码中硬编码
2. 环境变量命名统一遵循前缀+大写下划线规则，与YAML结构一一对应
3. 通用能力（重试、日志、告警）下沉至框架层，禁止每个用例重复实现

### 8.3 用例编写规范
1. 每个场景对应独立测试文件，命名清晰表意
2. 核心断言聚焦稳定性指标：成功率、事件丢失率、累积劣化情况
3. 失败场景必须留存完整请求响应快照，便于事后排查

### 8.4 设备操作边界（海康适配层）
海康门禁适配层默认对真实设备**只读（GET）**，仅在明确指示或确属必要时才写入（PUT），以规避误操作真机。

1. **默认只读，显式才写**：`HikvisionWorker` 默认只通过 GET 读取设备状态（工作状态 / 设备时间 / 门锁开启保持时间 openDuration 等），绝不主动修改设备配置。任何写入必须通过统一的 `--set KEY=VALUE` 通道显式声明；仅列在 `worker._DEVICE_WRITE_WHITELIST` 白名单内的键才会被 PUT，未列出的键一律忽略并标记 `device_write_unsupported`。新增可写能力只需在白名单登记一个键，无需新增 CLI 指令。
2. **必要情况例外（校时）**：恢复阶段的时间校准（`HEAL_TIME_SYNC`，即 `set_time` 校时）属于「必要情况」，即使未显式声明也会执行——前提是设备时间偏差超过 `time_skew_threshold`。注意海康 `PUT /ISAPI/System/time` 报文**必须包含 `timeMode` 与 `IANA` 字段**（否则返回 `MessageParametersLack / errorMsg=timeMode`），故 `set_time` 会先 GET 设备当前 Time 并原样保留这两个字段，仅替换 `localTime`（默认沿用设备 `timeZone`）。校时失败必须留痕，禁止静默吞异常。
3. **关门确认以门状态轮询为准**：`do_work` 远程开门后，通过轮询 `get_work_status().doorLockStatus`（映射：`[1]`=开门/解锁，`[0]`=闭锁/关闭）确认「已开→已关」全过程，再查询事件日志；查询时间窗口以开门时刻为锚点、后向小余量，终点取当前设备时间，避免猜测关门时刻或触碰设备事件 API 的「未来终点」限制。

### 8.5 前置条件就绪与事件链统计（门离线成因区分）
「门离线」有两种成因，必须区分处理，否则会把「前置条件没准备好」误判为「循环内设备故障」：

1. **循环前置条件未就绪（可自愈）**：门 1 离线可能是因为串口 1 的外设类型（`mode`）不对。在 `pre_loop_setup()` 开头执行 `_ensure_preconditions()`：
   - 若构造时给定 `required_serial_mode`（如 `externMode`/`readerMode`/`accessControlHost`/`accessDetection`，经 `--serial-mode` 传入），先 `GET /ISAPI/System/Serial/ports/1/capabilities` 校验该 mode 受支持；
   - 再 `GET /ISAPI/System/Serial/ports/1` 读取当前 `mode`；若不符，回写完整配置（`PUT`，仅替换 `mode`）切换外设类型；
   - 海康对串口配置切换返回 `statusCode=7 / subStatusCode=autoReboot`，表示设备将**自动重启**；`_ensure_preconditions()` 复用 3 阶段 `_wait_online`（离线→上线→连续确认）等待设备重新上线，再 `GET` 复核 `mode` 已变更；
   - 最终校验 `doorOnlineStatus==[1]`（门在线）。串口模式已就绪但门仍离线，则判为**设备真实故障**（前置条件失败）。
   - 若**未**配置 `required_serial_mode`，则只在门离线时判定前置条件失败（不自动切换，需人工介入），避免盲目改配置。
2. **循环过程中离线（设备问题）**：`do_work` 阶段远程开门被拒（如 `remoteDoorControlFailedDoorOffline`）属设备真实故障，由 `check()` 事实独裁判 `fail`；并在 `_chain_stats` 中记 `in_loop_offline`、在事实中附 `door_offline.in_loop=True`，与前置条件未就绪明确区分。
3. **事件链跨轮统计**：`HikvisionWorker` 累积 `_chain_stats`（轮数、触发/开门/关门命中数、`in_loop_offline`、`precond_fixed`、`precond_failed`），每轮 `check()` 汇总；示例在末尾打印事件链统计，便于观测整体稳定性与离线成因分布。
