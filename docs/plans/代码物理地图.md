## 引擎隔离约束验证(扫描结果与 AGENTS.md 对照)

✓ `harness/*` 仅 import `core.*`,无 `loop/` 或 `multi_agent/` 引用
✓ `loop/*` 仅 import `core.*` + 同引擎 `loop.*`,无 `multi_agent/` 引用
✓ `multi_agent/*` 仅 import `core.*` + 同引擎 `multi_agent.*`,无 `loop/` 或 `harness/` 引用
✓ `business/*` 跨引擎 import `core/harness/loop/multi_agent`,合法装配层

## 关键观察

1. **零内部依赖节点**(架构地基):`core/bus.py`、`core/voting.py`、`loop/context.py`、`loop/decision.py`、`loop/scheduler.py`、`multi_agent/adapter.py` — 这些模块构成各引擎的"地基",引擎隔离约束由它们支撑
2. **`harness/tracer.py` 特殊**:docstring 声明"仅依赖 core.bus",但实际无显式 import(通过 EventBus 实例注入工作,duck typing) — 一个"软依赖"
3. **`business/hikvision/worker.py` 1020 行**:除 examples/ 外最大单文件,值得考虑拆分
4. **空 `__init__.py`**:`harness/` 与 `multi_agent/` 的 `__init__.py` 都是 0 字节(与 core/loop/business 的 re-export 风格不一致)
5. **`runner.py` 是装配枢纽**:依赖 17 个内部模块,印证"business 是装配层"
