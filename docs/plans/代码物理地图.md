## 引擎隔离约束验证(扫描结果与 AGENTS.md 对照)

✓ `harness/*` 仅 import `core.*`,无 `loop/` 或 `multi_agent/` 引用
✓ `loop/*` 仅 import `core.*` + 同引擎 `loop.*`,无 `multi_agent/` 引用
✓ `multi_agent/*` 仅 import `core.*` + 同引擎 `multi_agent.*`,无 `loop/` 或 `harness/` 引用
✓ `business/*` 跨引擎 import `core/harness/loop/multi_agent`,合法装配层
✓ `business/hikvision/capabilities/*` 仅依赖同包 `scenario_schema` 与同目录文件,不反向 import 引擎层

## 关键观察

1. **零内部依赖节点**(架构地基):`core/bus.py`、`core/voting.py`、`loop/context.py`、`loop/decision.py`、`loop/scheduler.py`、`multi_agent/adapter.py` — 这些模块构成各引擎的"地基",引擎隔离约束由它们支撑
2. **`harness/tracer.py` 特殊**:docstring 声明"仅依赖 core.bus",但实际无显式 import(通过 EventBus 实例注入工作,duck typing) — 一个"软依赖"
3. **`examples/generic_harness.py` 1086 行**:除 business/ 外最大单文件,作为"通用装配模板"承担了示例 + 演示 + CLI 三重职责
4. **`business/hikvision/scenario_runner.py` 638 行**:当前领域装配枢纽(原 `runner.py` 已删除并被 `scenario_runner.py` 取代),依赖 14 个内部模块,印证"business 是装配层"
5. **空 `__init__.py`**:`harness/` 与 `multi_agent/` 的顶层 `__init__.py` 都是 0 字节(与 core/loop/business 的 re-export 风格不一致)
6. **`capabilities/` 是数据驱动场景层的核心**:`actions/*` + `preconditions/*` + `probes/*` 三类原子能力,新增用例 = 复制 `configs/scenario_template.yaml` 改字段,零代码改动
7. **`FakeScenarioAdapter` 与 `--dry-run` 已删除**:测试不变量禁止 fake/mock,真实设备验证走 `examples/scenario_run.py`,无 `HIK_HOST` 时 pytest 自动 skip 真机用例
