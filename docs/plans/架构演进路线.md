# 架构演进路线（Architecture Evolution Roadmap）

> 本文件记录 stability_harness_loop_multiagent 框架的**已知架构改进方向**，
> 目标是让框架"从架构上就是对的"——三引擎职责纯粹、边界可在模块层强制、
> 治理能力名副其实、控制循环事件驱动。
>
> 记录时间：2026-07-18
> 适用分支：`feat/stability-harness-loop-multiagent`
> 关联文档：`AGENTS.md`（架构约束与编码规范）

---

## 0. 现状研判

三引擎（harness / loop / multi_agent）职责划分已落实到位，且"互不 import"的架构底线被严格遵守。
经实测核查（2026-07-18），当前唯一的结构性瑕疵是：

- `loop/` 与 `multi_agent/` 实际都依赖 `harness.agent`（`Agent`/`AgentSpec`）
  与 `harness.bus`（`EventBus`）。
- `harness/bus.py` 仅用标准库；`harness/agent.py` 仅 `from .bus import EventBus`
  ——**两者自包含、零内部 harness 依赖**。
- 即"契约内核"仅有 2 个文件（bus + agent），却被放在 `harness/` 内，使 harness 同时扮演
  "契约内核"和"治理引擎"双重角色，导致：
  1. "三引擎对等"的叙事在物理依赖上被打破（harness 是事实底座）；
  2. 无法在模块层强制边界（loop/mas 名义只依赖"接缝"，但 import 的是整个 `harness` 包，
     静态上仍可能误 import 治理实现）。

另有两点"能力就绪但未接线"的问题：

- `harness/governance.py` 与 `harness/verify.py` 已完整实现并实测可用（8 项探针全过），
  但**未被 `runner.py` / `worker.py` / 示例接入，也无专门单测**——属"死代码"风险。
- `loop/driver.py` 的 `ControlLoop._run_round` 每轮**硬等** `max(recover_timeout, check_timeout)`
  秒，造成无效等待，并曾引发测试 `TimeoutError` 回归（已由调大 run_timeout 临时缓解，治本需事件驱动化）。

## 1. 改进方向一：契约内核 `core/` 拆分【P0 · 结构性 · 最高优先级 · 状态：已完成 2026-07-18】

> 已落地：`harness/bus.py`、`harness/agent.py` 迁入 `core/`；harness/loop/multi_agent/business/
> examples/tests 的 import 全部改写指向 `core`；顶层 `__init__` 仍 re-export 全部公共 API。
> 验证：`pytest tests/` 全绿（47 passed）、`examples/smoke.py` 通过、全仓 `harness.agent|harness.bus`
> 引用为 0。详见 `docs/plans/architecture-evolution/` 下的 design.md / plan.md / analysis-report.md。

### 目标
抽出一个独立包 `stability_harness_loop_multiagent/core/`，只放跨引擎共享的契约：

```
stability_harness_loop_multiagent/
├── core/                      # 新：契约内核（仅 bus + agent，零治理依赖）
│   ├── __init__.py            # re-export EventBus, Agent, AgentSpec
│   ├── bus.py                 # 原 harness/bus.py
│   └── agent.py               # 原 harness/agent.py
├── harness/                   # 治理引擎（消费 core 契约）
├── loop/                      # Loop 引擎（消费 core 契约）
└── multi_agent/               # MAS 引擎（消费 core 契约）
```

迁移后依赖关系变为三条对等边，全部只指向 `core`：

```
        core/  (bus + agent 契约内核)
        ↑    ↑     ↑
   harness   loop   multi_agent      ← 三引擎真正对等
```

### 迁移步骤（机械、低风险、对外 API 零影响）
1. 新建 `core/` 包，将 `harness/bus.py`、`harness/agent.py` 移入。
2. `core/__init__.py` re-export `EventBus, Agent, AgentSpec`（保持公共 API 不变）。
3. 批量替换 import：
   - `loop/**`：`from ..harness.agent` → `from ..core.agent`；`from ..harness.bus` → `from ..core.bus`
   - `multi_agent/**`：`from ...harness.agent` → `from ...core.agent`；`from ...harness.bus` → `from ...core.bus`
   - `harness/**`（runtime/watchdog/governance/verify/telemetry 内部对 agent/bus 的引用）：
     `from .agent` / `from .bus` → `from ..core.agent` / `from ..core.bus`
   - `business/**`、`examples/**` 中经 `__init__` re-export 的引用无需改动。
4. 顶层 `__init__.py` 维持现有 re-export（从 `core` 中转即可）。

### 验收标准
- `pytest tests/ -q` 全绿；`python examples/smoke.py` 正常。
- 静态校验：`grep -rn "harness.agent\|harness.bus" loop/ multi_agent/` 应为 0（即两引擎不再直接引用 harness 的契约文件）。
- 三引擎各自 import 中不再出现对**其他引擎治理实现**的引用（结构强制）。

### 收益 / 代价
- 收益：语义正确、依赖倒置、边界可被结构强制、单测更轻、为"多套 loop/mas 变体复用同一契约"铺路。
- 代价：一次约 10~12 处 import 的机械修改 + 文档更新；风险低。
- 判定：**对可复用框架是架构正确的形式，值得作为长期基线**（非过度设计，因成本可忽略、收益结构性）。

## 2. 改进方向二：Governance / Verify 真正接入【P1 · 状态：已完成 2026-07-18】

> 已落地：治理能力从"死代码"变为真正可用的能力。
> - `Governance.evaluate` 改为**两阶段**：先非突变探测全部策略，全部通过才提交 `Quota.consume`/`Budget.spend`；
>   拒绝时**不突变**配额/预算（修复"拒绝也扣额度"副作用 Bug）。
> - 新增 `gate_allowed(agent, req, timeout)` / `governance_decision(agent, req, timeout)` 异步
>   fail-closed 闸门（后者返回富回复含 `denied_ops`，集中 fail-closed 语义）。
> - `test_governance.py`（23 项）、`test_verify.py` 锁定 8 类探针行为；`test_hikvision_governance.py`、
>   `test_hikvision_governance_advanced.py`、`test_hikvision_verify.py` 验证端到端行为。
> - hikvision runner 以 **opt-in** 方式挂载 `GovernanceAgent`/`VerificationAgent`（与 Watchdog 同模式），
>   worker `act()` 入口发一次 `harness/govern/request`（`operation="round"`），纯总线契约不 import 治理实现。
> - **P1-b 校验真触发**：`HikvisionAdvisor` 解析计划后、采纳前发 `harness/verify/request`
>   （`enable_verify`，fail-closed 丢弃），让原本"挂载但未触发"的 `VerificationAgent` 真正生效。
> - **P1-c 熔断器真包裹**：worker 破坏性外部操作（开门/重启/基准重启）经 `_guarded_adapter_act`
>   受 `governance.breakers["hikvision-api"]` 保护——连续失败达阈值后熔断器打开、跳过调用，避免打爆设备。
> - **P1-d 按操作鉴权（维度化）**：治理新增 `DeniedOp(role?, capability?, op, match?)` 维度规则
>   （字符串配置自动归一为 `DeniedOp(op=...)`，向后兼容；`role`/`capability` 支持 `None` 与 `"*"` 通配；
>   `op` 支持 `match="exact"|"prefix"|"suffix"|"contains"|"regex"`，非法正则按未命中）；网关按 `role/capability/op` 维度匹配后
>   回复 `denied_ops`；worker 跳过被拒操作（如 `reboot`）仍执行其余操作（如 `remote_open_door`），无需把同步
>   `do_work` 重构为异步。
> - **治理结构化事实上报**：`Telemetry` 新增 `fact()`（kind="fact"，独立总线话题 `harness/fact/<name>`）；
>   治理网关决策点发 `governance.decision` 事实（allowed/reason/denied_ops/role/capability/operation/round），
>   调用方 fail-closed 超时路径由 worker 经 `governance.telemetry` 补发，二者不重复。
> - **治理观测面板（GovernancePanelAgent）**：新增 Observer（opt-in）订阅 `harness/fact/governance.decision`，
>   聚合为可读 dashboard（放行/拒绝分布、按角色/能力/操作的拒绝计数、超时 fail-closed 计数、轮次覆盖），
>   并提供 `timeseries()` 零依赖时间序列视图（累计放行/拒绝/fail-closed + 每轮被拒操作数）；面板响应
>   `governance/panel/request` 回发 `governance/panel`（内含聚合 `panel` 与 `timeseries`）供真实拉取，
>   演示见 `examples/governance_panel_demo.py`（用拉取的 `timeseries` 生成 plotly 交互式趋势报告，未装 plotly 时
>   回退零依赖 CSV；同时演示 exact/prefix/suffix/contains/regex 五种匹配）。runner 挂载治理网关时一并
>   `governance.telemetry=tel`（使事实真正发到总线）并把面板加入返回字典 `gov_panel`。观测面板可见
>   「哪一轮、为何被拒、按操作拒绝哪些」。`denied_ops` 返回仍为操作名列表（worker 仅消费操作名）。
> - 拒绝语义：fail-closed **只拦操作、不 halt 循环**（`emit_abort` 网关默认 `False`；硬违例 halt 作为可选开关）。

### 现状
`Governance`（AccessControl + Quota + Budget + CircuitBreaker）与 `Verifier`（输入护栏 + run_eval 聚合）
已实现并实测可用，未接入业务链路、无单测。**（P1 已解决：现已接总线、有单测）**

### 接入原则
**必须走总线接缝**，绝不破坏"三引擎互不 import"约束：
- 治理话题遵循 `harness/<domain>/request` 约定：`harness/govern/request`、`harness/verify/request`。
- loop/mas 只发 request 并 await reply，不 import `governance`/`verify` 实现（worker 仅用 `self.request` + 话题字符串）。

### 具体接线点
- **Governance**：`HikvisionWorker.act()` 在 `to_thread(do_work)` 之前发 `harness/govern/request`
  （`{role, capability:"door-test", operation:"round"}`），被授予才执行本轮全部操作；外部 API 调用统一受
  `CircuitBreaker` 保护（Governance 内部持有）。
- **Verifier**：runner 以 opt-in 方式挂载 `VerificationAgent(bus, verifier)`，可在后续按需对 LLM 输出做 fail-closed 护栏。
- **Watchdog / Governance 共用 `harness/abort` 接缝**：仅当 `emit_abort=True` 时治理违规才发布 `harness/abort`
  中止循环（网关默认 `False`，只拦操作）。

### 验收标准（均已达成）
- 新增 `tests/test_governance.py`、`tests/test_verify.py`：固化 8 项探针行为（AccessControl / Quota / Budget /
  CircuitBreaker / evaluate 拒绝不突变 / Verifier fail-closed / run_eval / 总线网关）。
- 端到端（`test_hikvision_governance.py`）：注入拒绝策略 → worker 跳过 do_work 并上报 `governance_denied`；
  配额未误扣。
- 全量 `pytest` 绿 + `examples/smoke.py` 通过，无回归。

## 3. 改进方向三：ControlLoop 事件驱动化【P2 · 状态：已完成 2026-07-18】

> 已落地：`_run_round` 改为"收齐 `target/recovered` + `target/checked` 即返回 + 超时兜底"
> （`asyncio.wait(ALL_COMPLETED)` + `ensure_future` + `cancel` 兜底）。
> **P2-b**：`_collect_votes` 同样事件驱动——收齐投票后（静默期 `vote_settle` 内无新票）即返回，
> 无投票者时仍于 `vote_timeout` 上限内终止（防死锁不变量不变）。新增 `vote_settle` 参数。
> 实测全量测试从 **~195s 降至 ~32s**（约 160s+ 无效硬等被消除）。
> 防死锁不变量不变（超时仍为硬上限）。
> 见 `tests/test_control_loop_event_driven.py`（`_run_round` 早到/永不回复 + `_collect_votes` 早到/无投票者）。

### 现状
`ControlLoop._run_round` 每轮固定 `await asyncio.sleep(max(recover_timeout, check_timeout))`，
即使早已收齐 `target/recovered` / `target/checked`，仍硬等至超时才结束。
**（P2 已解决：改为事件驱动 + 超时兜底）**

### 方案
- 改为"**收齐即返回**"：用 `asyncio.wait` + `ALL_COMPLETED` 等待"所有预期回复到达"或"超时兜底"两者先到。
- 真机环境仍以超时作为存活兜底（保留防死锁不变量）。
- 配合 `Scheduler` 退避/抖动，整体循环节奏由"事件 + 超时"共同驱动，而非固定睡眠。
- 投票收集（`_collect_votes`）保留 `vote_timeout` 超时兜底（投票为尽力而为，按设计不为其引入投票者计数耦合）。

### 收益（已验证）
- 测试大幅加速（无需为每个 round 等待整段超时）：全量 pytest 195s → 33s。
- 真机无效等待减少，循环响应更及时。
- 消除超时回归的根因。

### 验收标准（均已达成）
- `test_control_loop_event_driven.py`：回复早到 → 单轮耗时远小于 `max(recover,check)` 超时；
  无投票者 → 循环仍在 `vote_timeout` 上限内终止。
- 全量 `pytest` 绿 + `examples/smoke.py` 通过，无回归。

## 4. 改进方向四（可选）：依赖抽象化【P3 · 状态：暂缓（按需，YAGNI）】

> **暂缓判定（2026-07-18）**：P3 仅在"需要替换总线实现（跨进程/网络总线、内存测试替身）"时才具 ROI。
> 当前三引擎已通过 `core.bus` 契约内核解耦，且无任何"替换总线"的真实需求，按 YAGNI 原则暂缓，
> 不提前抽象。一旦需求出现再按本方向推进。

- 用 `typing.Protocol` 定义 `BusProtocol`，让 `loop` / `multi_agent` 依赖抽象接口而非具体 `EventBus` 类。
- `Agent` 是实现继承（含生命周期），纯抽象化较难，可暂留具体基类。
- 仅当未来出现"需要替换总线实现（如跨进程/网络总线）"的需求时再推进。

## 5. 执行顺序与优先级

| 优先级 | 方向 | 性质 | 风险 |
|--------|------|------|------|
| P0 | `core/` 契约内核拆分 | 结构性重构 | 低（机械） |
| P1 | Governance / Verify 接入 + 单测 | 补齐能力 + 测试 | 中（需设计总线话题） |
| P2 | ControlLoop 事件驱动化 | 行为优化 | 中（影响时序，需保防死锁） |
| P3 | 依赖抽象化（Protocol） | 进阶 | 低（可选） |
| P4 | 用例集驱动（数据驱动场景层） | 扩展能力 | 低（纯增量） |

建议顺序：**先 P0（打地基，让边界纯粹）→ 再 P1（治理能力落地，含单测锁定行为）→ 后 P2（消除时序缺陷）→ P3 按需 → P4 把 108 条用例沉淀为 YAML**。

## 6. （新增）用例集驱动：数据驱动场景层【P4 · 扩展能力 · 状态：已完成 2026-07-18】

> 需求来源：《门禁对讲通用稳定性用例集》（108 条）暴露出的核心诉求——**每加一条用例就改一次 worker 代码**不可持续。
> 目标：把用例抽象成一份 YAML（target + stress + probe + loop），**新增加例 = 新增 YAML，零代码改动**，
> 与 AGENTS.md「扩展只改一个引擎」「每次扩展只改一个引擎」的约束完全契合。

### 6.1 落点（全部在 `business/hikvision/`，领域层，不触碰三引擎）

- `scenario_schema.py`：`Scenario` / `Target` / `Stress` / `Probe` / `LoopCfg` 数据类 + `from_yaml` /
  `from_dict` + **校验** + **环境变量插值** `${VAR}` / `${VAR:-default}` + **字段路径解析**
  `resolve_field`（点号分段 + 列表下标，哨兵表达「不存在」）+ `compare_probe`（equals / in / 数值宽松比较）。
- `scenario_adapter.py`：
  - `ScenarioISAPIAdapter`：真实设备适配器（**唯一适配器**；`FakeScenarioAdapter` 与 `--dry-run` 已删除，
    测试不变量禁止 fake/mock），复用 `HikvisionClient` 的 Digest 鉴权与新增的通用 `request_json`；
    `reboot` 走 `client.reboot()`，`upgrade`/`issue` 按 `stress.endpoint`/`method`/`body` 发请求，
    重启类操作后轮询探测端点等待上线（离线→上线两阶段）。
- `scenario_worker.py`：`ScenarioWorker(WorkerAgent)`——每轮 `do_work`（执行 stress + observe 探测）
  → `recover`（在线已在 adapter 内等待）→ `check`（解析字段、与期望值断言、产出事实字典）。
  - **事实独裁**：`probe_ok=False` → `fail`；`probe_na=True`（存在性字段缺失）→ 不判失败。
  - **早停**：截止时间 `loop.deadline`（NT，未测试）与 `stop_on_na` 经总线 `harness/abort` 安全中止循环，
    复用既有 ExternalAbortStop 契约（不新增跨引擎耦合）。
- `capabilities/`（**原子化能力,数据驱动场景层的核心抽象**）：把"重启 / 升级 / 下发 / 长巡"类操作
  拆成三类原子能力,被 `ScenarioWorker` 装配调度,新增用例零代码改动：
  - `actions/`：`RebootAction` / `UpgradeAction` / `RemoteOpenAction` / `QueryEventsAction` /
    `DispatchAction` / `SleepAction` / `NoopAction` / `SwitchSerialAction`
  - `probes/`：`FieldProbe` / `OnlineProbe` / `CountProbe` / `EventChainProbe`
  - `preconditions/`：`BaselineRecordPrecondition` / `DeviceOnlinePrecondition` / `SerialModePrecondition`
  - `__init__.py` 工厂入口：`create_action` / `create_probe` / `create_precondition`（按 type 字符串分发）
- `scenario_runner.py`：`run_scenario(scenario, *, adapter, ...)` 把一份 Scenario 组装成完整
  `ControlLoop`（复用 harness/loop 引擎），返回结构化汇总（轮数 / 裁决分布 / NA 计数 / 早停原因 / 遥测）。
- `examples/scenario_run.py`：CLI——`--scenario <yaml> [--rounds N] [--interval S]`，
  直接跑任意用例；敏感信息走环境变量。
- `configs/`：`stability_0001_reboot.yaml`（重启）、`stability_0009_wired_network.yaml`（长巡 + 截止 + NA）、
  `scenario_template.yaml`（复制即用的模板）。

### 6.2 与 108 条用例的映射

| 用例族（文档子模块） | stress.type | probe 形态 | loop 形态 |
|---|---|---|---|
| 重启稳定性（0001-0004） | `reboot` | 重启后 GET 状态、字段==预期 | max_rounds=N、interval=reboot_interval |
| 升级稳定性（0005-0008_0001） | `upgrade` | 同上 | 同上 |
| 网络连接 / 硬件稳定性（0009-0047） | `none` | 长巡 GET、字段∈{online/connect/…} | max_rounds 大 + deadline + na_if_absent |
| 下发稳定性（0048-0106） | `issue` | 下发后查状态正常、未重启 | max_rounds=1（或按需） |

> 每条用例的「预期结果」被统一表达为 `probe.field + expect_equals/expect_in (+ na_if_absent)`；
> 「测试数据」被表达为 `stress.*` 与 `loop.*`；「前置条件/不通过/不涉及」被表达为事实独裁 + NA 早停。
> **新增用例无需改代码，仅写 YAML。**
