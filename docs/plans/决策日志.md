## 5 个关键架构决策

### 决策 1 · P0 契约内核 core/ 拆分(2026-07-18 · 已完成)

- **问题**: `loop/` 与 `multi_agent/` 实际依赖 `harness.agent` / `harness.bus`,导致 harness 同时扮演"契约内核"和"治理引擎"双重角色,"三引擎对等"叙事在物理依赖上被打破
- **方案**: 抽出独立包 `core/`,把 `harness/bus.py` + `harness/agent.py` 迁入;三引擎 import 全部改写指向 `core`;顶层 `__init__` re-export 保持对外零影响
- **代价**: 约 10-12 处 import 机械修改 + 文档更新,风险低

### 决策 2 · P1 Governance / Verify 真正接入(2026-07-18 · 已完成)

- **问题**: `harness/governance.py` 与 `harness/verify.py` 完整实现但未接入 runner/worker,属"死代码"风险
- **方案**: `Governance.evaluate` 改两阶段(拒绝不突变配额);新增 `gate_allowed` / `governance_decision` 异步 fail-closed 闸门;hikvision runner opt-in 挂载;worker `act()` 入口发 `harness/govern/request`(纯总线契约不 import 治理实现);配套 P1-b/c/d(校验真触发 / 熔断器真包裹 / 按操作鉴权维度化) + `GovernancePanelAgent` 治理面板
- **代价**: 增加总线话题与策略匹配维度(`DeniedOp` 5 种 match)的维护成本

### 决策 3 · P2 ControlLoop 事件驱动化(2026-07-18 · 已完成)

- **问题**: `ControlLoop._run_round` 每轮固定 `await asyncio.sleep(max(recover_timeout, check_timeout))`,即使早已收齐事件仍硬等至超时(曾引发测试 `TimeoutError` 回归)
- **方案**: 改为"收齐即返回"——`asyncio.wait` + `ALL_COMPLETED` 等待"所有预期回复到达"或"超时兜底"先到;`_collect_votes` 同样事件驱动(静默期 `vote_settle` 内无新票即返回);真机环境仍以超时作为存活兜底
- **代价**: 影响时序,需保防死锁(超时仍为硬上限);新增 `vote_settle` 参数;补 `test_control_loop_event_driven.py` 锁定行为
- **收益**: 实测全量测试从 ~195s 降至 ~32s(约 160s+ 无效等待消除)

### 决策 4 · P3 依赖抽象化(Protocol)(2026-07-18 · 暂缓,YAGNI)

- **问题**: `loop` / `multi_agent` 依赖具体 `EventBus` 类而非抽象接口,未来替换总线实现(跨进程/网络)成本高
- **方案**: 用 `typing.Protocol` 定义 `BusProtocol`,让 `loop` / `multi_agent` 依赖抽象接口而非具体类
- **代价**: 当前三引擎已通过 `core.bus` 契约内核解耦,且无"替换总线"真实需求,按 YAGNI 原则暂缓,不提前抽象

### 决策 5 · P4 数据驱动场景层(2026-07-18 · 已完成)

- **问题**: 《门禁对讲通用稳定性用例集》108 条用例暴露"每加一条用例就改一次 worker 代码"不可持续,与 AGENTS.md"扩展只改一个引擎"约束不契合
- **方案**: 把用例抽象成 YAML(`target` + `stress` + `probe` + `loop` 四段),落点全部在 `business/hikvision/`(`scenario_schema.py` / `scenario_adapter.py` / `scenario_worker.py` / `scenario_runner.py`);新增用例 = 新增 YAML,零代码改动
- **代价**: 增加领域层 schema 校验与字段路径解析复杂度;`HikvisionClient` 需增量新增 `request_json`;需维护 YAML 模板与示例配置文件

## 5 个核心不变量(架构的安全底线)

### 不变量 1 · 三引擎互不 import(引擎隔离)

- **含义**: `harness/` `loop/` `multi_agent/` 互不直接 import,所有跨引擎通信只走 `EventBus`
- **守护机制**: `tests/test_harness_regression.py` 静态守护;P0 后 `grep -rn "harness.agent\|harness.bus" loop/ multi_agent/` 应为 0;`core/` 契约内核让三引擎在模块层真正对等

### 不变量 2 · 防死锁(超时为硬上限)

- **含义**: 每个跨引擎 `await` 都有超时 + 确定性兜底;`max_rounds` / `max_duration` 硬上限保证循环一定终止;`recheck` 次数上限为 1;ControlLoop 事件驱动(收齐即返回 + 超时兜底)
- **守护机制**: `tests/test_control_loop_event_driven.py` 验证回复早到 → 单轮耗时远小于超时;测试用极短超时(`vote_timeout=0.1`、`recover_timeout=0.05`);全量 `pytest` 绿验证循环终止

### 不变量 3 · 事实独裁(安全底线)

- **含义**: `DecisionAuthority` 唯一裁决权——任一事实为 `False` → 本轮 `fail`,不可被风险分或投票翻转为 `pass`;风险分只能加 `warn` / `recheck` 注解;`critical` 事件强制 `recheck`;决策出错 → 保守 `warn(60)`(`CONSERVATIVE_RISK`),绝不乐观 `pass`
- **守护机制**: `tests/test_stability_harness_loop_multiagent_smoke.py::test_decision_authority_fact_dictatorship_unit` 单元锁定;`tests/test_scenario.py` 验证 `probe_ok=False` → `fail`、`probe_na=True` → 不判失败;`HikvisionAdvisor` 只投 `(risk, confidence)` 票,不能直接影响 verdict

### 不变量 4 · 治理拒绝 fail-closed(只拦操作不 halt 循环)

- **含义**: 治理网关超时 / 拒绝只拦操作、不 halt 循环(`emit_abort` 默认 `False`);worker 跳过被拒操作(如 `reboot`)仍执行其余操作(如 `remote_open_door`),保证测试连续性
- **守护机制**: `tests/test_governance.py`(23 项) + `tests/test_verify.py`(8 项) + `tests/test_hikvision_governance*.py` 端到端断言:注入拒绝策略 → worker 跳过 do_work 并上报 `governance_denied`、配额未误扣

### 不变量 5 · 拒绝不突变配额 / 预算

- **含义**: `Governance.evaluate` 两阶段——先非突变探测全部策略,全部通过才提交 `Quota.consume` / `Budget.spend`;拒绝时不突变配额/预算(修复"拒绝也扣额度"Bug),避免被拒操作白白消耗限流配额与预算资源
- **守护机制**: `tests/test_governance.py` 固化"evaluate 拒绝不突变"探针;端到端 `tests/test_hikvision_governance.py` 断言注入拒绝策略后配额未误扣;治理结构化事实 `governance.decision` 经 `Telemetry.fact()` 发到 `harness/fact/governance.decision` 话题,可被 `GovernancePanelAgent` 聚合观测
