# 海康门禁设备稳定性测试框架设计方案
## 基于三引擎自治循环框架的智能自愈与多智能体演进

本方案基于通用自治循环框架 `stability-harness-loop-multiagent` 进行业务层适配，用于实现海康门禁设备具备“自主诊断与自愈能力”的无人值守长稳测试。

---

## 1. 架构映射与“真·多智能体”自治自愈设计

为了充分发挥多智能体的自治性，我们拒绝单一的“脚本化执行”，而是将**测试执行、异常诊断、策略决策、自愈恢复**拆解为多个协同工作的智能体。

```
                    ┌────────────────────────┐
                    │      ControlLoop       │ ◄──────┐
                    │ (事实独裁 + 决策判断)  │        │
                    └───────────┬────────────┘        │
                                │ loop/tick           │
                                ▼                     │
                    ┌────────────────────────┐        │
                    │   LLM Worker Agent     │        │
                    │ (动作执行 + 事实收集)  │        │
                    └─────┬────────────▲─────┘        │
             target/      │            │              │
             checked      │            │ 诊断/恢复     │ agent/
             (Facts)      │ 异常触发   │ 指令         │ vote/reply
                          ▼            │              │ (加权风险分)
                    ┌──────────────────┴─────┐        │
                    │  LLM Diagnostic Agent  │        │
                    │   (异常分析与自愈者)   │        │
                    └───────────┬────────────┘        │
                                │                     │
                                ▼                     │
                    ┌────────────────────────┐        │
                    │   LLM Advisor Agent    │ ───────┘
                    │ (自然语言指令评估)     │
                    └────────────────────────┘
```

### 智能体角色定义与自治职责：

| 智能体名称 | 核心职责 | 自治性体现 (LLM / 规则驱动) |
|---|---|---|
| **LLM Worker Agent** | 执行测试指令（如开门、重启），收集基础事实（设备状态、事件是否存在）。 | **动作自治**：解析 `STABILITY_INSTRUCTION`，根据当前设备状态动态决定下一步操作（而非硬编码顺序）。 |
| **LLM Diagnostic Agent** | **自愈核心**：当 Worker 报告异常事实（如“事件未找到”）时介入，分析原因并执行自愈操作。 | **自愈自治**：当发生异常时，自动分析事实（例如比对设备时间与系统时间，检测网络延迟），生成诊断方案（如“执行校时后重新触发/重新查询”），执行自愈并反馈结果。 |
| **LLM Advisor Agent** | 评估长周期运行是否符合用户自然语言定义的期望。 | **规则自治**：评估趋势（如重启时间逐渐变慢）和隐性要求，对整体风险进行打分。 |
| **ControlLoop** | 协调生命周期，判定最终 Verdict，执行连续失败熔断。 | **控制自治**：提供事实独裁安全网，确保即使 LLM 尝试自愈失败，也能安全终止测试。 |

## 2. 核心自愈流程设计 (异常自治处理)

以用户提到的**“查询不到事件”**以及**“设备突然掉线”**为例，展示多智能体的协作自愈机制：

```
       [Worker Agent]                [Diagnostic Agent]             [设备/网关]
             │                               │                           │
   1. 触发开门且查询事件失败                  │                           │
   (event_recorded = False)                  │                           │
             │─────── 报告异常事件 ─────────>│                           │
             │    (agent/incident)           │                           │
             │                               │ 2. 诊断：检查设备时间     │
             │                               ├──────────────────────────>│
             │                               │ <────── 返回时间 ─────────│
             │                               │                           │
             │                               │ 3. 判定：设备时间不正确   │
             │                               │ 4. 自愈：下发校时指令     │
             │                               ├──────────────────────────>│
             │                               │ <────── 校时完成 ─────────│
             │                               │                           │
             │                               │ 5. 重新验证：通知重新触发 │
             │<────── 重新验证/查询 ─────────┤                           │
   6. 重新执行开门并查询                      │                           │
   (若成功，标记为 "self_healed")            │                           │
             │                               │                           │
```

### 常见异常的自治自愈策略：

#### 场景 A：查询不到事件
* **诊断步骤**：
  1. 检查设备时间与测试机系统时间偏差是否大于阈值（如 $\pm 3$ 秒）。
  2. 若偏差大，说明存在时钟漂移，导致按时间段查事件落空。
* **自愈动作**：
  1. 调用 ISAPI 对设备进行强制校时。
  2. 通知 Worker 重新下发一次协议指令，并重新查询时间窗口内的事件。
  3. 若重新查询成功，将事实标记为 `True`，但在 `RoundRecord` 中备注 `self_healed: time_sync`，作为警告记录。

#### 场景 B：设备突然掉线
* **诊断步骤**：
  1. 触发 ping 检测。如果 ping 通，说明仅 ISAPI 服务卡死或网络端口被防火墙拦截；如果 ping 不通，说明物理掉线。
  2. 查询局域网内其他设备状态，确认是否是测试机自身断网。
* **自愈动作**：
  1. 如果是测试机断网，静默等待网络恢复。
  2. 如果仅被测设备 ISAPI 端口不响应但 ping 可通，尝试下发硬重启或等待服务守护进程拉起。
  3. 若掉线时间超过 `max_recover_timeout`，触发熔断退出，留存掉线时段的抓包和日志快照。

## 3. 关键设计细节澄清

### 3.1 为什么重启在线探活需要“连续 2 次成功”？
* **原因**：嵌入式设备在重启期间，网卡驱动和网络服务（如 HTTPS 服务）可能在就绪边缘出现**“抖动”**。
  - 现象：第一次连接可能由于端口刚 bind 成功而短暂握手，但由于内部服务尚未初始化完毕或立刻发生二次暖启动，该连接随即断开。
  - 后果：如果只查 1 次成功就判定为“已恢复上线”，紧接着下发业务指令（如开门）就会遇到 `ConnectionResetError` 导致用例误报失败。
* **改进**：支持通过配置项 `probe_confirm_count` 进行动态调整。用户可将其设为 `1`（快速判断，适合高性能/高稳定性设备）或 `2`（保守判定，默认值）。

### 3.2 探测性重启的条件触发
* **规则**：**只有在 RunConfig 中声明了重启步骤，或者测试用例本身属于“重启稳定性”子模块时**，才在循环前准备阶段执行探测性重启测量 `baseline_recover_time`。
* **实现**：`conftest.py` 会检查用例的 `pytest.mark` 或 `test_config` 中是否启用了重启操作。若未启用（如纯开门测试），则跳过探测性重启，避免无意义的等待。

## 4. 环境变量 `STABILITY_INSTRUCTION` 自然语言解析机制

我们利用 LLM 的 Prompt Engineering，将自然语言转化为智能体的执行约束：

### 系统 Prompt 输入结构 (LLM Advisor / Worker)：
```
你是一个门禁设备测试诊断专家。用户向你下达了稳定性测试的自然语言要求：
"{{ STABILITY_INSTRUCTION }}"

请将其转化为：
1. Worker 执行时的偏好控制（例如：开门后是否需要等待、等待多久）。
2. Diagnostic Agent 的自愈白名单（例如：允许校时自愈、允许重试）。
3. 判定劣化风险的阈值规则（用于 Advisor 打分）。
```

### 示例解析效果：
* **用户输入**：`STABILITY_INSTRUCTION="测试频繁开门，开门后如果 2 秒内查不到事件，就立刻校时重试一次。重启不要测。"`
* **LLM 解析为结构化指令**：
  - `run_reboot`: `False` (跳过探测性重启与重启操作)
  - `event_check_delay`: `2s`
  - `auto_heal_actions`: `["time_sync_and_retry"]`
  - `eval_rules`: "If reboot detected -> fail; If event delay > 2s -> warn"

## 5. 配置中心更新 (YAML & Env)

新增自愈及 LLM 节点配置：

```yaml
device:
  host: "192.168.1.100"
  port: 80
  username: "admin"
  password: "password123"
  http_timeout: 5

loop:
  mode: "count"
  total_rounds: 1000
  round_interval: 2
  consecutive_failure_threshold: 10
  max_recover_timeout: 180
  probe_interval: 5
  probe_confirm_count: 2         # 探活确认次数，默认为 2，可设为 1
  warmup_time: 60

event:
  door_open_delay: 3
  query_retry: 3
  query_retry_interval: 5

autonomy:
  enable_self_healing: true      # 开启自治自愈
  diagnostic_llm_model: "gemini-3.5-flash" # 用于异常诊断的LLM
  instruction: ""                # 对应 STABILITY_INSTRUCTION 环境变量
```
